# Production Implementation Stack

This document turns `production-readiness.md` into an implementation stack. It separates what is already implemented, what can be added with the current stack, and what needs external approval, source validation, or partner/internal APIs.

## Target Architecture

Keep one project MCP server, `fdn-data`, as the stable interface for the LLM.

```text
LLM / Skills
  -> fdn-data MCP tools
    -> source adapters
      -> public APIs, partner APIs, internal data, local rule tables
    -> normalization layer
    -> cache and freshness metadata
    -> response contract
```

The LLM should never call raw external URLs directly for production answers. It should call a named MCP tool whose response includes source, period, freshness, assumptions, and non-recommendation guardrails.

## Stack Decisions

### 1. MCP/API Layer

Use the existing Python standard-library MCP server for the hackathon/MVP. For production, keep the MCP contract but split implementation into source adapters:

- `finlife_adapter.py`
- `ecos_adapter.py`
- `market_proxy_adapter.py`
- `housing_market_adapter.py`
- `policy_rules_adapter.py`
- `cashflow_adapter.py`
- `benchmark_adapter.py`

Recommended production runtime:

- Python MCP server for continuity with the current code.
- SQLite only for local fixtures and tests.
- PostgreSQL or managed relational DB for cached normalized source data.
- Redis or equivalent cache for API rate limiting and short-lived freshness cache.
- Versioned JSON/YAML rule tables for policy, tax, and eligibility assumptions.

### 2. Data Contract

Every production tool should return:

```json
{
  "source": "...",
  "source_url_or_provider": "...",
  "as_of": "...",
  "freshness_status": "fresh|stale|fixture|user_assumption|unavailable",
  "input": {},
  "values": [],
  "assumptions": [],
  "decision_use": "...",
  "not_recommendation_reason": "..."
}
```

This should be enforced in tests so the LLM cannot present stale or assumption-based values as exact facts.

## Domain Stack

### Home Ownership

Production need:

- Regional home price history.
- Transaction volume.
- Jeonse/monthly rent levels.
- Home type normalization.
- Region code normalization.
- Loan product rates.
- LTV/DSR/policy-loan constraints.
- Acquisition tax, brokerage, moving, maintenance cost assumptions.

Recommended tools:

| Tool | Data source type | Priority | Notes |
|---|---|---|---|
| `get_home_market_snapshot(region, home_type, contract_type, ym_range)` | Public real-estate transaction API or internal housing dataset | P0 | Core missing piece for home decisions |
| `get_rent_or_jeonse_snapshot(region, home_type, ym_range)` | Public jeonse/monthly rent API or internal housing dataset | P0 | Needed for buy vs rent vs wait |
| `normalize_region_code(region_text)` | Local region-code table | P0 | Required before public API calls |
| `get_finlife_rent_house_loan_rate_range(...)` | Finlife rent-house loan API | P1 | Extends current Finlife adapter |
| `get_policy_loan_constraints(user_profile, home_profile, as_of)` | Versioned policy rule table | P1 | Should not be generated by LLM |
| `estimate_home_transaction_costs(region, price, home_type, as_of)` | Versioned tax/fee rule table | P1 | Returns assumptions and effective date |

Implementation order:

1. Region-code normalization.
2. Public or internal transaction/rent adapter.
3. Finlife rent-house loan wrapper.
4. Rule-table adapter for LTV/DSR/tax/fees.
5. Scenario composition tool combining market, loan, macro, and user numbers.

- Pension/IRP/product terms.
- Contribution limits and tax benefits.
- KRW-based proxy series.

- Spending floor and healthcare reserve assumptions.

Recommended tools:

| Tool | Data source type | Priority | Notes |

| `get_asset_proxy_returns(symbols, currency, years)` | Alpha Vantage, exchange data, cached market data | P0 | Generalizes current monthly series |

Implementation order:

2. Add FX series through ECOS or market adapter.
3. Add pension/tax rule tables with effective dates.
4. Add product-term adapter only after legal/compliance wording is settled.

## Cross-Cutting Production Requirements

### Caching

- Finlife product data: cache by service, product type, `dcls_month`, page.
- ECOS key stats: cache by service and cycle.
- Alpha Vantage: cache by symbol and latest monthly date because free-tier limits are low.
- Housing market data: cache by region code, home type, contract type, month.
- Rule tables: version by effective date and source.

### Observability

Track:

- Tool called.
- Source provider.
- Cache hit/miss.
- Freshness status.
- Missing input that triggered reverse question.
- Whether response used API data, fixture, or user assumption.

Do not log API keys, raw account identifiers, or sensitive transaction details.

### Testing

Add tests for:

- Every MCP tool returns the production data contract.
- Web-search-free behavior for API-backed tasks.
- Freshness/status labels.
- Fixture parity with live-compatible wrappers.
- Region normalization.
- Policy rule version selection by date.
- Consent denial path.
- AX reverse-question shape: exactly four options, fourth is `직접 입력`.
